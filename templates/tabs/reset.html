⚠️ Full Platform Reset — this cannot be undone

The Full Platform Reset below wipes everything across all three phases. Use the component resets above it to redeploy a single component without touching the cluster.

Phase 1 — K8s Nodes
kubeadm reset
kubelet + containerd removed
All K8s + Longhorn packages
/etc/kubernetes, /var/lib/etcd
/var/lib/longhorn, /opt/cni
~/.kube + cluster-artifacts
SSH authorized_keys wiped
/etc/hosts entries removed
Phase 2 — GitLab VM
gitlab-ctl stop
apt-get remove --purge gitlab-ce
/etc/gitlab deleted
/var/opt/gitlab deleted
/var/log/gitlab deleted
/opt/gitlab deleted
OAuth tokens invalidated
Registry data removed
Phase 3 — Controller
generated/inventory.ini
generated/group_vars/all.yml
gitlab-inventory.ini
gitlab-vars.yml
gitlab-outputs.json
jupyterhub-vars.yml
dashboard-vars.yml
entire generated/ dir
⚠ After a full reset you must re-run the Bootstrap tab with SSH passwords to re-establish trust — all authorized_keys are wiped.

📓 Reset JupyterHub

Uninstalls the JupyterHub Helm release and deletes the jhub namespace — removes all secrets, ConfigMaps, and OAuth credentials. Clears the generated vars file.

Note: Longhorn volumes use a Retain reclaim policy — existing user data PVs remain in Longhorn until manually deleted. Re-run the JupyterHub tab to redeploy.

⚙ Reset Admin Dashboard

Stops and removes the workbench-admin systemd service, deletes the unit file, and removes /etc/workbench/ which contains the GitLab admin token. Clears the generated vars file.

The ~/k8s-launcher repo clone on the control plane is preserved — it is the running codebase, not a deployment artifact. Re-run the Admin Dashboard tab to redeploy.
